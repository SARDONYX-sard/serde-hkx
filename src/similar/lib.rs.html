1//! This crate implements diffing utilities.  It attempts to provide an abstraction
2//! interface over different types of diffing algorithms.  The design of the
3//! library was once inspired by pijul's diff library by Pierre-Étienne Meunier
4//! and also inherits the patience diff algorithm from there.
5//!
6//! The API of the crate is split into high and low level functionality.  Most
7//! of what you probably want to use is available top level.  Additionally the
8//! following sub modules exist:
9//!
10//! * [`algorithms`]: This implements the different types of diffing algorithms.
11//!   It provides both low level access to the algorithms with the minimal
12//!   trait bounds necessary, as well as a generic interface.  See the
13//!   module's **"Which algorithm should you use?"** section for practical
14//!   recommendations beyond the default [`Algorithm::Myers`].
15//! * [`udiff`]: Unified diff functionality.
16//! * [`utils`]: utilities for common diff related operations.  This module
17//!   provides additional diffing functions for working with text diffs.
18//!
19//! # Sequence Diffing
20//!
21//! If you want to diff sequences generally indexable things you can use the
22//! [`capture_diff`] and [`capture_diff_slices`] functions.  They will directly
23//! diff an indexable object or slice and return a vector of [`DiffOp`] objects.
24//!
25//! ```rust
26//! use similar::{Algorithm, capture_diff_slices};
27//!
28//! let a = vec![1, 2, 3, 4, 5];
29//! let b = vec![1, 2, 3, 4, 7];
30//! let ops = capture_diff_slices(Algorithm::Myers, &a, &b);
31//! ```
32//!
33//! If your sequence is computed on demand, or if you want to diff by a derived
34//! key, wrap it in [`algorithms::CachedLookup`]. This materializes each item at most
35//! once and then exposes it through normal indexing.
36//!
37//! # Text Diffing
38//!
39//! Similar provides helpful utilities for text (and more specifically line) diff
40//! operations.  The main type you want to work with is [`TextDiff`] which
41//! uses the underlying diff algorithms to expose a convenient API to work with
42//! texts:
43//!
44//! ```rust
45//! # #[cfg(feature = "text")] {
46//! use similar::{ChangeTag, TextDiff};
47//!
48//! let diff = TextDiff::from_lines(
49//!     "Hello World\nThis is the second line.\nThis is the third.",
50//!     "Hallo Welt\nThis is the second line.\nThis is life.\nMoar and more",
51//! );
52//!
53//! for change in diff.iter_all_changes() {
54//!     let sign = match change.tag() {
55//!         ChangeTag::Delete => "-",
56//!         ChangeTag::Insert => "+",
57//!         ChangeTag::Equal => " ",
58//!     };
59//!     print!("{}{}", sign, change);
60//! }
61//! # }
62//! ```
63//!
64//! ## Trailing Newlines
65//!
66//! When working with line diffs (and unified diffs in general) there are two
67//! "philosophies" to look at lines.  One is to diff lines without their newline
68//! character, the other is to diff with the newline character.  Typically the
69//! latter is done because text files do not _have_ to end in a newline character.
70//! As a result there is a difference between `foo\n` and `foo` as far as diffs
71//! are concerned.
72//!
73//! In similar this is handled on the [`Change`] or [`InlineChange`] level.  If
74//! a diff was created via [`TextDiff::from_lines`] the text diffing system is
75//! instructed to check if there are missing newlines encountered
76//! ([`TextDiff::newline_terminated`] returns true).
77//!
78//! In any case the [`Change`] object has a convenience method called
79//! [`Change::missing_newline`] which returns `true` if the change is missing
80//! a trailing newline.  Armed with that information the caller knows to handle
81//! this by either rendering a virtual newline at that position or to indicate
82//! it in different ways.  For instance the unified diff code will render the
83//! special `\ No newline at end of file` marker.
84//!
85//! ## Bytes vs Unicode
86//!
87//! Similar module concerns itself with a looser definition of "text" than you would
88//! normally see in Rust.  While by default it can only operate on [`str`] types,
89//! by enabling the `bytes` feature it gains support for byte slices with some
90//! caveats.
91//!
92//! A lot of text diff functionality assumes that what is being diffed constitutes
93//! text, but in the real world it can often be challenging to ensure that this is
94//! all valid utf-8.  Because of this the crate is built so that most functionality
95//! also still works with bytes for as long as they are roughly ASCII compatible.
96//!
97//! This means you will be successful in creating a unified diff from latin1
98//! encoded bytes but if you try to do the same with EBCDIC encoded bytes you
99//! will only get garbage.
100//!
101//! # Ops vs Changes
102//!
103//! Because very commonly two compared sequences will largely match this module
104//! splits its functionality into two layers:
105//!
106//! Changes are encoded as [diff operations](crate::DiffOp).  These are
107//! ranges of the differences by index in the source sequence.  Because this
108//! can be cumbersome to work with, a separate method [`DiffOp::iter_changes`]
109//! (and [`TextDiff::iter_changes`] when working with text diffs) is provided
110//! which expands all the changes on an item by item level encoded in an operation.
111//!
112//! As the [`TextDiff::grouped_ops`] method can isolate clusters of changes
113//! this even works for very long files if paired with this method.
114//!
115//! # Deadlines and Performance
116//!
117//! For large and very distinct inputs the algorithms as implemented can take
118//! a very, very long time to execute.  Too long to make sense in practice.
119//! To work around this issue all diffing algorithms also provide a version
120//! that accepts a deadline which is the point in time as defined by an
121//! [`Instant`] after which the algorithm should give up.  What giving up means
122//! depends on the algorithm.  For instance due to the recursive, divide and
123//! conquer nature of Myers' diff you will still get a pretty decent diff in
124//! many cases when a deadline is reached.  Whereas on the other hand the LCS
125//! diff is unlikely to give any decent results in such a situation.
126//!
127//! The [`TextDiff`] type also lets you configure a deadline and/or timeout
128//! when performing a text diff.
129//!
130//! Note that on wasm targets calling [`Instant::now`] will result in a panic
131//! unless you enable the `wasm32_web_time` feature.  By default similar will
132//! silently disable the deadline checks internally unless that feature is
133//! enabled.
134//!
135//! # Feature Flags
136//!
137//! The crate by default does not have any dependencies however for some use
138//! cases it's useful to pull in extra functionality.  Likewise you can turn
139//! off some functionality.
140//!
141//! * `std`: enabled by default.  Turns on integration with the Rust standard
142//!   library (including deadline support via [`Instant`]).  Disabling this
143//!   feature makes the crate `no_std` compatible (with `alloc`).
144//! * `text`: this feature is enabled by default and enables the text based
145//!   diffing types such as [`TextDiff`].
146//!   If the crate is used without default features it's removed.
147//! * `unicode`: when this feature is enabled the text diffing functionality
148//!   gains the ability to diff on a grapheme instead of character level.  This
149//!   is particularly useful when working with text containing emojis.  This
150//!   pulls in some relatively complex dependencies for working with the unicode
151//!   database.
152//! * `bytes`: this feature adds support for working with byte slices in text
153//!   APIs in addition to unicode strings.  This pulls in the
154//!   [`bstr`] dependency.
155//! * `inline`: this feature gives access to additional functionality of the
156//!   text diffing to provide inline information about which values changed
157//!   in a line diff.
158//! * `serde`: this feature enables serialization to some types in this
159//!   crate.  For enums without payload deserialization is then also supported.
160//! * `wasm32_web_time`: this feature swaps out the use of [`std::time`] for
161//!   the `web_time` crate.  Because this is a change to the public interface,
162//!   this feature must be used with care.  The instant type for this crate is
163//!   then re-exported top-level module.
164//! * `hashbrown`: in `no_std` mode, switches internal map storage from
165//!   [`alloc::collections::BTreeMap`] to `hashbrown::HashMap`.
